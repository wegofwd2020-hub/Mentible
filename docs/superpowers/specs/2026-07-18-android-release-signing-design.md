# Android release signing ‚Äî design

**Date:** 2026-07-18
**Status:** Approved (brainstorm), pending plan
**Issue:** #328 (APK debug-signing)
**Branch:** `chore/android-release-signing` (off `main`)

## 1. Problem

The distributed Android app is signed with the **React Native template debug keystore** ‚Äî a
publicly-known key (`android/app/debug.keystore`, alias `androiddebugkey`, SHA-1
`5E:8F:16:06:2E:A3:CD:2C:4A:0D:54:78:76:BA:A6:F3:8C:AB:F6:25`, CN=Android Debug) that Expo prebuild
copies into every project. `mobile/android/app/build.gradle` sets `buildTypes.release.signingConfig =
signingConfigs.debug` (with the "In production, generate your own keystore" caution left in place).
Verified: the released `mentible-0.2.2-vc13` APK is signed with this key (apksigner: CN=Android Debug).

**Consequence (authenticity gap):** anyone can sign a malicious APK with the *same* public key, so an
Android device accepts it as a legitimate "Mentible" update. There is effectively no signing integrity
on the distributed app ‚Äî the exact risk #328 flags.

## 2. Goals / non-goals

**Goals**
- G1. Sign release builds with a **private** key that only the maintainer holds.
- G2. Keep the key and its passwords **out of git** (public repo) ‚Äî no secret ever committed.
- G3. Keep dev / CI builds working without the release key (they don't need it).
- G4. Ship a re-release signed with the new key, with the mandatory one-time-reinstall communicated.
- G5. Document the signed-release build process and the **backup requirement**.

**Non-goals**
- N1. No CI-based signing / APK-build workflow (releases stay local; CI does not build the APK ‚Äî the
  Mobile CI job is typecheck/lint/test only). Structure does not preclude adding it later.
- N2. No Play Store / Play App Signing enrollment (distribution is sideloaded APK via the landing page).
- N3. No rotation of the existing debug key's *history* ‚Äî prior releases stay as they are; the cutover
  is forward-only.

## 3. Design

> **This is a managed Expo workflow.** `mobile/.gitignore:6-7` ‚Äî "Native projects ‚Äî generated by
> `expo prebuild`; not committed." `android/` is regenerated from `app.json` + config plugins, so a
> manual `build.gradle` edit is (a) not committable and (b) lost on the next prebuild. The signing
> config is therefore injected by a **committed Expo config plugin**, and the keystore lives **outside**
> the regenerated `android/` so a `prebuild --clean` can't wipe it.

### 3.1 The keystore (custody: local file, outside android/)
Generate a private release keystore, non-interactively, with strong random passwords:
`mobile/credentials/mentible-release.keystore` ‚Äî RSA 2048, validity 10000 days (~27 yr), alias
`mentible`. `mobile/credentials/` is **gitignored** and outside `android/` (survives prebuild). The
generation writes the passwords **directly into the gitignored config** (¬ß3.2) and does **not** echo the
private key or passwords to the transcript/logs.

**Custody:** the keystore + passwords are the maintainer's to hold **forever**. Losing either means the
app can never be updated again (a new key = a new signature = every user must reinstall). Back both up
out-of-band immediately ‚Äî this is a hard handoff, not an implementation detail.

### 3.2 Secrets out of git
- `mobile/credentials/keystore.properties` (**gitignored**) ‚Äî `storeFile` (absolute or
  `mobile/`-relative path to the keystore), `storePassword`, `keyAlias`, `keyPassword`. The only place
  the passwords live in the tree.
- `mobile/credentials/keystore.properties.example` (**committed**) ‚Äî the same keys with placeholder
  values, documenting the required shape + where to put the real file.
- `.gitignore` ‚Äî add explicit `mobile/credentials/` (except the `.example`), plus `*.keystore`/`*.jks`
  belt-and-braces.

### 3.3 Expo config plugin (the durable injection)
`mobile/plugins/withReleaseSigning.js` (**committed**) ‚Äî a config plugin using `withAppBuildGradle` that,
at prebuild time, patches the generated `app/build.gradle` idempotently to:
- add a `signingConfigs.release` that loads `mobile/credentials/keystore.properties` **if it exists**
  (reading `storeFile`/passwords/alias from it);
- set `buildTypes.release.signingConfig` to `signingConfigs.release` **when the properties file exists**,
  else leave the debug fallback.

The plugin patches by string-anchor (idempotent: no-op if its marker is already present), so re-running
prebuild is safe. Registered in `app.json` `plugins` (alongside `expo-router`, `expo-secure-store`).

Rationale: committable + survives regeneration (G1/G3); a maintainer with the credentials present builds
a real-key release, while a fresh clone / CI (no credentials) still gets a working debug-signed build ‚Äî
no secret hardcoded, and the injection can't silently vanish on a prebuild.

### 3.4 Re-release (vc14)
Bump `versionCode` to 14 (in `app.json` ‚Äî the source of truth). Release recipe (now explicitly includes
prebuild so the plugin applies):
`cd mobile && npx expo prebuild -p android` ‚Üí `cd android && ANDROID_HOME=‚Ä¶ ./gradlew :app:assembleRelease`
(real-key signed, credentials present) ‚Üí publish to **mambakkam-net** as asset `Mentible.apk`, tag
`mentible-0.2.2-vc14`. Release notes MUST carry:

> ‚öÝ **One-time reinstall required.** This build is signed with a new key, so it cannot install over a
> previous version. Uninstall the old Mentible app first, then install this one. (Future updates will
> install normally.)

### 3.5 Documentation
A short `docs/android-release-signing.md` (or a section in an existing release doc): the signed-release
build steps, the `keystore.properties` shape, the **backup requirement**, and the one-time-reinstall
migration note for reference.

## 4. Verification

- `apksigner verify --print-certs` on the new APK shows the **new private cert** (NOT CN=Android Debug,
  NOT SHA-1 `5E:8F:16:06:‚Ä¶`).
- After `expo prebuild`, the generated `app/build.gradle` contains the injected `signingConfigs.release`
  and the release buildType points at it (the plugin ran).
- A release build **with** `mobile/credentials/keystore.properties` present succeeds and is real-key
  signed.
- A prebuild+build **without** the credentials still succeeds (debug fallback) ‚Äî proves G3 (dev/CI/fresh
  clone unaffected); the plugin's presence alone changes nothing without the properties file.
- `git status` shows tracked additions are ONLY: `mobile/plugins/withReleaseSigning.js`,
  `mobile/credentials/keystore.properties.example`, `.gitignore` entries, the signing doc, `app.json`
  (plugin registration + versionCode 14). **No** keystore or real `keystore.properties` tracked
  (`git ls-files --error-unmatch` on each must error).
- The landing download (`mambakkam-net/releases/latest/download/Mentible.apk`) resolves to vc14.

## 5. Risks

- **R1. Keystore loss = permanent lockout.** Mitigation: the plan's final step is an explicit backup
  handoff to the maintainer; the app cannot recover from a lost key. This is inherent to Android signing,
  not specific to this design.
- **R2. Existing testers must reinstall.** Accepted (chosen "migrate now" ‚Äî small early base; cost grows
  over time). Mitigated by the explicit release-notes instruction.
- **R3. Passwords in the working tree.** They live only in the gitignored
  `mobile/credentials/keystore.properties`; verification asserts it is untracked. Generation must not
  echo them to logs.
- **R4. Config-plugin string-anchor fragility.** The plugin patches `build.gradle` by matching an anchor
  string in the Expo template; a future Expo upgrade could change that template and make the patch a
  silent no-op. Mitigation: the plugin throws if its anchor is not found (fail loud, not silent), and
  the ¬ß4 post-prebuild assertion (build.gradle contains the injected block) catches a regression.
