## User Story

> **As a** self-learner power user
> **I want** to type any email address and send the lesson I just generated as a PDF attachment
> **so that** I can share what I learned with a study partner, mentor, or my own secondary inbox without screenshots or copy-paste.

---

## Background

The lesson view already supports PDF export (per the prerequisite). This ticket adds a sibling action: **Email this lesson** — opens a small modal, the user types a destination email and an optional personal message, taps Send, and the backend emails the same PDF as an attachment via a transactional email service.

Email-share is **on-demand only** — no scheduling, no auto-send, no recurring delivery. Each send is one explicit user action with a typed recipient.

This is the first outbound surface the product has. It introduces an abuse vector (harassment-via-repeated-sends, spam to non-consenting recipients) not covered by the current `PARAMETERS.md §5`. See *Cross-references* below.

## Acceptance Criteria (Gherkin)

### AC1 — Trigger from lesson view

```
Given the user is viewing a generated lesson
When they tap the share menu and choose "Email this lesson"
Then a modal opens with two fields:
  - Recipient email (required)
  - Personal message (optional, ≤ 500 chars)
And a Send button (disabled until recipient is RFC-valid).
```

### AC2 — RFC 5322 recipient validation

```
Given the email modal is open
When the user types an invalid email (no @, malformed domain, etc.)
Then the Send button stays disabled
And an inline error appears: "Enter a valid email address".

When the user types a syntactically valid email
Then the Send button becomes enabled.
```

### AC3 — Optional personal message

```
Given the email modal is open
When the user types into the personal message field
Then the character count updates in real time
And typing past 500 characters is hard-stopped at 500.

When the user leaves the field empty
Then Send still works
And the email body falls back to a default "User shared this lesson with you" preamble.
```

### AC4 — Backend endpoint

```
Given the mobile app is sending an email-share
When it calls POST /share/email
Then the request body has shape:
  {
    "lesson_id": "<uuid>",
    "recipient": "<rfc5322 email>",
    "message": "<optional, ≤ 500 chars>"
  }
And the backend responds 202 Accepted on success
And 400 on validation failure
And 429 on rate-limit hit.
```

### AC5 — Rate limit

```
Given the user has sent 5 email-shares today (UTC midnight rollover)
When they attempt a 6th
Then the backend returns 429
And the mobile app shows: "Daily share limit reached (5/day). Try again tomorrow."
```

### AC6 — Email body content

```
Given a successful send
Then the recipient receives an email with:
  - Subject: "<lesson title> — shared via StudyBuddy Q"
  - Body containing (in order):
      1. The user's personal message (if any), quoted
      2. A one-line provenance: "This lesson was generated by the sender using StudyBuddy Q."
      3. A link to the StudyBuddy Q AUP
      4. A short footer: "If you didn't expect this, ignore it. The sender's email address is [the user's email if available — see Privacy below]."
  - PDF attachment: the lesson rendered to PDF via the existing pipeline.
```

### AC7 — Audit log without lesson body

```
Given a successful send
Then the backend writes an audit row containing:
  - sha256(lesson_id + send_timestamp)
  - sha256(recipient_email)
  - sender_user_id
  - timestamp
  - status (sent | bounced | rejected)
And NEVER:
  - the raw lesson body
  - the raw recipient email
  - any portion of the user's Anthropic API key (per ADR-001 — defensive belt-and-braces; key isn't used in this flow)
And the audit row is retained 30 days.
```

### AC8 — Provider-side bounce handling

```
Given the recipient address bounces hard at the provider level
When the bounce webhook fires
Then the audit row's status updates to "bounced"
And the sender sees a non-blocking notification on next app open: "Your email to <recipient masked> bounced."
```

### AC9 — No content stored after send

```
Given a successful send
When 60 seconds have elapsed
Then the lesson body, recipient email, and personal message are absent from any backend persistent store
(Audit log retains only hashes per AC7.)
```

### AC10 — Unsubscribe / report-abuse footer

```
Given the recipient receives the email
Then the email footer contains:
  - A "report this as spam" mailto link (routes to support@<domain>)
  - A note: "This is a one-off share, not a mailing list — there is nothing to unsubscribe from."
```

## Edge cases

- **Invalid email format** — covered by AC2; client-side validation prevents the request.
- **Recipient bounces** — covered by AC8; surfaced asynchronously via webhook.
- **Network failure mid-send** — mobile retries once, then surfaces "Send failed, try again."
- **Rate-limit hit** — covered by AC5; client-side counter as hint, server-side enforcement is authoritative.
- **Spam-flagged sender** — if the transactional email provider flags the sender domain, fall back to a queue + admin alert; user sees "Send delayed, will deliver shortly."
- **Recipient is the sender's own email** — allow it (legitimate self-archive use case).
- **Repeated sends to same recipient** — counts toward the 5/day cap; future ticket may add a "max 2/recipient/day" sub-cap if abuse observed.

## Out of Scope

- Sharing via link / public URL (separate v2+ candidate per SCOPE.md:199)
- Multi-recipient sends (single recipient per send at MVP)
- Scheduled or recurring sends
- Email templates beyond the AC6 fixed body
- Sender identity verification (DKIM/SPF setup is engineering's call, not in spec)
- Reply handling — replies to the share email go to a no-reply address; conversation routing is v2+
- Tracking opens / click-throughs (privacy + scope)
- iOS support (Android-first per D3)
